# 总体原则

> 适用：所有语言、所有项目

1. 【建】只在确定有性能问题时才优化代码。过早优化是万恶之源；"看起来快"的源码未必生成更快的目标码，只会难以理解和维护。(源:建议1)
2. 【规】风格指南未覆盖之处，参照现有代码的做法保持一致。(源:规则1)
3. 【建】始终在最高级别编译警告下编译代码，警告当错误对待。(源:建议3)
4. 【建】不使用难懂的技巧性语句，除非确有必要并加注释。如 `*pPos++ += 1` 应拆为两条语句。(源:建议5)
5. 【建】关系紧密的代码应尽可能相邻。(源:建议4)
6. 【规】禁止魔数：避免不易理解的数值，用有名字的常量/枚举替代。(源:规则31/57)
7. 【建】一行只写一个语句；一条语句只声明一个变量。(源:建议32/规则61)
8. 【规】行宽 ≤120 字符；长表达式在低优先级操作符处换行，操作符放新行之首。(源:规则47，裁决取120)
9. 【规】缩进：新代码使用 4 空格；存量文件维持其现有风格，不强制改造。(源:规则36+附录D，裁决)
