# /vflow:commit — 智能提交

$ARGUMENTS

## 流程

### 阶段 1：仓库校验

1. 确认在 git 仓库内、无 rebase/merge 冲突进行中
2. `git status` + `git diff`（已暂存与未暂存）获取全部改动

### 阶段 2：改动分类 [required]

把脏文件分为两类，**绝不静默提交第二类**：

- **本次会话相关**：本会话编辑过的文件 + 关联的任务档案（`.vflow/tasks/`）
- **无法识别**：本会话没动过的文件（可能是用户手工改动或并行工作）→ 列出并询问是否纳入，默认排除

### 阶段 3：拆分建议

按关注点聚类（源码 / 测试 / 文档 / 任务档案）。若检测到多组独立变更（>300 行且跨多个顶级目录），建议拆成多次提交并给出分组方案；用户不接受则单次提交。

### 阶段 4：生成提交信息

遵循项目约定（中文提交信息）：

```
类型: 简短描述

- 要点（可选，动机/影响范围）

任务: <vflow 任务ID，如 06-12-roundness-algo（有活动/刚归档任务时）>
```

类型：feat/fix/refactor/docs/test/chore。首行 ≤72 字符。

**脱敏检查**：扫描暂存内容中的 token/key/password/secret 模式，发现疑似泄密 → 停止并警告。

### 阶段 5：一次性确认 [HARD STOP]

展示完整方案（文件清单 + 提交信息），等待用户回复：
- "ok" / "可以" / "行" → 执行
- "我自己来" / "manual" → 中止，不再生成第二版方案
- 其他反馈 → 按反馈调整后重新展示

### 阶段 6：执行

```bash
git add <明确的文件列表>   # 禁止 git add -A
git commit -m "<信息>"
git log --oneline -1       # 回显确认
```

## 铁律

1. 不提交无法识别的文件，除非用户明确同意
2. 不使用 --no-verify / --force；钩子失败 → 修复问题后新建提交，不用 --amend
3. 不推送（push 需用户另行要求）
4. 一次提交只做一件事
